🧮 Core Business Logic: Smart Transporter Selection

When an administrator initiates a transport request, the OrderService dynamically reads through the condition matrix layer. It queries the data registry, filters out expired parameters, evaluates fuel adjustment calculations, and isolates the specific transporter offering the **absolute lowest contract price** for that unique destination city.

Algorithm Implementation Strategy:
public async Task<Transporter?> GetCheapestTransporterForDestinationAsync(int cityId, decimal weight)
{
    var validConditions = await _context.Conditions
        .Include(c => c.Contract)
        .ThenInclude(con => con.Transporter)
        .Where(c => c.CityId == cityId && c.Contract.IsActive && c.Contract.EndDate >= DateTime.UtcNow)
        .ToListAsync();

    return validConditions
        .OrderBy(c => c.CalculateFormulaPrice(weight))
        .Select(c => c.Contract.Transporter)
        .FirstOrDefault();
}

🔄 The 10-Step Order Lifecycle Flow

Step 1: Contract Initialization

Administrator registers Transporters and uploads active fixed pricing arrays per City.

Step 2: Order Entry Creation

Administrator enters client requests for a shipping assignment.

Step 3: Best Price Routing Engine Execution

System automatically selects the lowest cost transporter via internal LINQ conditions.

Step 4: SMTP Relay Notification Dispatch

System acts as an email relay, dispatching secure job offers directly to the selected transporter.

Step 5: Carrier Truck Registration Placement

Transporter logs in and enters the assigned truck registration plate numbers into the tracking form.

Step 6: Status Promotion to Approved

Once plate criteria values validate, order records change status fields from Pending to Approved.

Step 7: Automated SLA Workers Guard

An internal background worker monitoring the queue triggers an expiration if a carrier fails to assign a plate registration within a strict 2-day SLA window.

Step 8: Auto-Fallback Re-routing Execution

Upon an SLA expiration, the system flags the order back to Pending, stripes the current assignment, and automatically assigns the next lowest-priced transporter.

Step 9: Plant Loading Bay Dispatch Verification

The physical truck enters the loading zones; dispatch note CSV files compile directly into the application context.

Step 10: Lifecycle Process Closure

The platform reads the incoming manifests, completes order updates, and converts the workflow state to Finished.

🛠️ Resolved Architectural & Engineering Challenges

  • 1. Optimizing Complex Dropdown Overloads (150+ Duplicate Records Slimmed to 8 Distinct Cities)
    Problem: The destination dropdown parsed every individual database record node, over-allocating HTML DOM structures.
    Solution: Rewrote components inside PopulateOrderDropdowns() to query the unique city enum keys utilizing programmatic grouping logic: .GroupBy(d => d.City).Select(g => g.First())
  • 2. Managing High Volume Relational AutoMapper Graph Mappings
    Problem: Upgrading system components from legacy configurations to newer core infrastructure profiles introduced entity dependency injection exceptions.
    Solution: Reconfigured decoupled dependency setups, ensuring explicit profile registries bind accurately across multi-level domain entities during compilation execution.
